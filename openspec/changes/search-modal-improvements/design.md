## Context

現在の検索モーダル（`SearchModal.ts`）には、以下の技術的課題があります。
1. **フィードバックの欠如**: 検索やページングの通信中、タイトル横に小さな `...` が表示されるだけで、結果リストがいつ更新されるのかが不明瞭。
2. **多重リクエスト**: 通信中に入力フィールドやボタンが有効なままであるため、連打による重複リクエストが可能。
3. **イベントのリーク**: モーダル上でのクリックイベントが背面の DOM に伝搬し、`Navigator` による画像移動をトリガーしてしまう。

## Goals / Non-Goals

**Goals:**
- 検索結果エリアを覆う視覚的なローディング・オーバーレイの実装。
- 通信中の入力フォーム（input, button）の無効化。
- モーダル内からのイベント伝搬の完全な遮断。
- 既存の `comic-helper-spinner` アニメーションの再利用。

**Non-Goals:**
- 検索アルゴリズムやアダプターのロジック変更。
- モーダル自体のレイアウトやデザインの大幅な刷新。

## Decisions

### 1. 検索結果エリアのラッパー構造の導入
検索結果セクション（`resultsSection`）を `relative` ポジションのコンテナで包み、その中に `absolute` ポジションの `spinnerOverlay` を配置します。
- **理由**: モーダル全体をロックするのではなく、結果エリアのみを視覚的にブロックすることで、何が更新されているかを明確にします。

### 2. `SearchModalComponent.setUpdating` の機能拡張
単に `...` を表示するだけでなく、以下の処理を行うように拡張します。
- `spinnerOverlay` の表示/非表示の切り替え。
- `input` 要素および `submitBtn` 要素の `disabled` 属性の制御。
- **代替案**: グローバルな `LoadingIndicator` を使用することも検討しましたが、検索モーダルというコンテキスト内に限定した方がユーザー体験が向上すると判断しました。

### 3. イベント伝搬阻止の徹底
以下の要素に対して `e.stopPropagation()` を確実に適用します。
- `form` の `submit` イベント。
- `pagination` ボタンの `click` イベント。
- `content` 要素全体への `click` イベントハンドラ追加（セーフティネットとして）。

### 4. `UIManager._performSearch` の修正
`silent` フラグに関わらず、すべての検索開始時に `setUpdating(true)` を呼び出すように変更します。

## Risks / Trade-offs

- **[Risk]** オーバーレイの表示中にモーダルを閉じると、状態が不安定になる可能性がある。
  - **Mitigation**: `onClose` 時には通信のキャンセルは行わない（現在の Store ベースの設計を維持）が、UI コンポーネントが破棄されるため問題はない。
- **[Trade-off]** フォームを `disabled` にすると、通信中にキーワードを修正できなくなる。
  - **Rationale**: 短時間の通信であるため、多重送信防止のメリットが勝ると判断。
