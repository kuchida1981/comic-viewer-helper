## Context

現在のコードベースでは、UIテキストの多くがコンポーネント内に英語または日本語でハードコードされています。また、`shortcuts.js` では機能の識別を「表示名（Label）」で行っているため、表示名を日本語に変更すると `main.js` のキー判定ロジックが壊れるという密結合の状態にあります。

## Goals / Non-Goals

**Goals:**
- 言語判定ロジック (`src/i18n.js`) の導入と、`en` / `ja` 辞書の構築。
- ショートカット定義の ID ベース化（表示テキストからの分離）。
- 全コンポーネントからのハードコードテキストの排除。

**Non-Goals:**
- 言語切り替えボタンの実装（今回はブラウザ言語による自動判定のみ）。
- 日本語、英語以外の言語の追加（構造として対応可能にするが、翻訳データは含めない）。

## Decisions

### 1. `src/i18n.js` の新設と `t()` 関数の提供
- **決定**: オブジェクトベースの辞書を持ち、ドット記法（例: `ui.spread`）でアクセスできる `t` 関数をエクスポートする。
- **理由**: 将来的に辞書を外部JSON化したり、パラメータ埋め込み機能を拡張したりしやすいため。
- **フォールバック**: 未定義のキーや未対応の言語に対しては、常に `en` のテキストを返すようにする。

### 2. ショートカット定義の ID 化
- **決定**: `src/shortcuts.js` の各要素に `id` プロパティを追加し、`main.js` では `sc.label === 'Next Page'` ではなく `sc.id === 'nextPage'` で判定するように変更する。
- **理由**: 翻訳によってラベル名が変わっても、ロジックの堅牢性を維持するため。

### 3. 言語判定ロジック
- **決定**: `navigator.language` を取得し、最初の2文字（例: `ja-JP` -> `ja`）を使用して辞書を選択する。
- **理由**: 最も標準的で軽量な判定方法であるため。

## Risks / Trade-offs

- **[Risk] 翻訳漏れによる UI の欠落**
  - → **Mitigation**: 開発時に `npm run check-types` 等を活用し（可能であれば辞書キーの型定義を行い）、フォールバックロジックで空文字にならないように保証する。
- **[Trade-off] ファイル分割によるオーバーヘッド**
  - → **Mitigation**: Vite によるビルドプロセスでバンドルされるため、ユーザー実行時のパフォーマンスへの影響は無視できる。
