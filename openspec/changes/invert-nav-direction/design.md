## Context

閲覧対象の書籍画像は右開き（右から左へ進む）が大半であり、現在の実装では `<` が「前へ」、`>` が「次へ」という左開き書籍を前提とする方向割り当てが行われている。キーバインドも同様に `ArrowRight` が「次へ」で、読者の直感に反している。この設計では、方向割り当てを反転し、将来の設定画面（#73）で方向を切り替えられるよう定数として管理する。

## Goals / Non-Goals

**Goals:**
- GUIボタンの表示文字列と動作のマッピングを反転する
- キーバインドの `ArrowRight`/`ArrowLeft` を反転する
- ナビゲーション方向の割り当てを定数として集約し、再利用可能にする

**Non-Goals:**
- `j`/`k`, `PageUp`/`PageDown`, `Space`/`Shift+Space` の動作変更
- 設定画面 UI の実装（#73 で対応予定）
- ホイールナビゲーション方向の変更（ホイールは上下スクロールの直感に基づく）

## Decisions

### Decision 1: 方向定数の導入箇所
**選択: `src/shortcuts.js` に `NAV_DIRECTION` 定数を定義し、キーの割り当てとGUIボタンの動作の両方がこれを参照する**

- 代替案A: 各ファイルで個別に反転ロジックを書く → 変更箇所が分散し、将来の設定画面对応で漏れが出る可能性が高い
- 代替案B: 新規の dedicated 定数ファイルを作成する → この規模の変更では過度なファイル分割になる
- `shortcuts.js` はキーバインド定義の中心で、既にナビゲーション方向の拠点になっている。定数をここに置くことで参照先を最小化できる

### Decision 2: キー割り当ての変更スコープ
**選択: `ArrowLeft`/`ArrowRight` のみを移動し、他のキーは変更しない**

- `j`/`k` はVim由来で「下へ進む」という方向性を持ち、上下スクロールの直感と一致している
- `PageDown`/`PageUp`, `Space`/`Shift+Space` も同様に上下の直感に基づく
- `ArrowLeft`/`ArrowRight` のみが右開き書籍の読み方向と直接対応するため、反転の対象とする

### Decision 3: GUIボタンの実装
**選択: `NavigationButtons.js` の `configs` 配列で `onPrev`/`onNext`・`onFirst`/`onLast` の割り当てを反転する**

- 表示文字列（`<`, `>` など）は変更しない。ボタンの視覚的な左右配置も変えない
- あくまで「この文字列がどの動作に対応するか」のマッピングを反転するだけで、UIの見た目は維持する

## Risks / Trade-offs

- [Risk: 既存ユーザーの混乱] → 方向が変わることで既存ユーザーが一時的に混乱する可能性がある。将来の設定画面（#73）で方向を選択可能にすることで緩和を予定する
- [Risk: ホイールナビゲーションとの不整合] → ホイールの上下は変えないため、ホイールと矢印キーで方向が異なる組み合わせが発生する。ただしホイールは「上下スクロール」としての直感があるため、これは許容し既存の動作を維持する
- [Trade-off: 定数導入のコスト] → 小さな変更に定数を導入するは若干の間接性を加える。ただし将来の設定画面対応で明確にペイする
