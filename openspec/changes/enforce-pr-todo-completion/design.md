## Context

PRの本文に記載されたタスクが完了したかどうかを、マージ前に機械的にチェックする仕組みが必要です。

## Goals / Non-Goals

**Goals:**
- PRの説明文にあるTODOリスト（`- [ ]`）を自動検出し、未完了があればCIを失敗させる。
- PRの作成時および編集時にリアルタイムでフィードバックを提供する。

**Non-Goals:**
- コメント欄のTODOチェック。
- 外部ツール（Jira等）との同期。

## Decisions

### 1. 実装手段: GitHub Actions (shell script)
- **理由**: 特別な外部Actionを導入せずとも、標準の `github.event.pull_request.body` をシェルスクリプトでパースするだけで実現可能で、軽量かつ安全。
- **実装詳細**: 
  単純な検索による誤検知を避けるため、以下の2段階のフィルタリングを行う。
  1. **コードブロックの除去**: `sed` を使用して、三連バッククォート（`` ``` ``）で囲まれた範囲をスキャン対象から除外する。
  2. **行頭マッチングの厳密化**: 正規表現を使用して、行頭（またはインデント後）にある `- [ ]` のみを検知対象とする。
  
  ```bash
  # 実装イメージ
  CLEAN_BODY=$(echo "$BODY" | sed '/^```/,/^```/d')
  if echo "$CLEAN_BODY" | grep -qE '^([[:space:]]*)?- \[ \]'; then
    exit 1
  fi
  ```

### 2. トリガー: pull_request (opened, edited, synchronize)
- **理由**: PRが作成された時だけでなく、TODOをチェックして説明文を更新した際（edited）や、修正コミットを積んだ際（synchronize）にも再チェックが必要なため。

## Risks / Trade-offs

- **[Risk]**: 意図しないテキストが TODO として誤検知される（例：コードブロック内の `- [ ]`）。
  - **Mitigation**: 上述の通り、`sed` によるコードブロック除去と行頭限定の正規表現マッチングを組み合わせることで、誤検知を最小限に抑える。
- **[Risk]**: PR本文が巨大な場合のパフォーマンス。
  - **Mitigation**: シェルスクリプトの `sed` および `grep` は非常に高速であり、GitHub Actions の実行時間への影響は無視できる。
